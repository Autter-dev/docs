Autter sits at the merge layer of your CI/CD pipeline and runs automated checks before any code reaches your main branch. Catch issues, generate missing tests, and document your code — all without changing your existing workflow.

## Pipeline configuration

Autter pipelines are configured in a single YAML file at `.autter/pipeline.yml`. Use built-in templates as a starting point, then customize enforcement levels to match your team's standards.

```yaml
# .autter/pipeline.yml
template: security-audit
config:
  severity_threshold: medium
  block_on_findings: true
  notify:
    - slack: "#security-alerts"
    - email: security@yourcompany.com
```

You can also configure per-rule severity levels directly:

```yaml
# autter.config.yml — customise enforcement levels
rules:
  security:
    severity: block          # prevent merge
  performance:
    severity: warn           # comment but allow merge
  conventions:
    severity: info           # informational only
  deprecated_apis:
    severity: block
    exceptions:
      - path: "legacy/**"    # known legacy code, don't block
```

## Available pipeline templates

Autter ships with four templates that cover the most common pre-merge workflows:

<Tabs>
  <Tab title="Standard review">
    The default template. Runs code quality checks, convention detection, and test coverage analysis on every PR. Best for teams getting started with automated review.

    Checks included:
    - Bug and logical error detection
    - Convention drift from your merge history
    - Test coverage for changed files
    - PR summary generation
  </Tab>
  <Tab title="Security audit">
    Runs security-focused scanners with configurable severity thresholds. Integrates with Slack and email for immediate notification on findings.

    Checks included:
    - Dependency vulnerability scanning
    - Injection risk detection (SQL, XSS, command injection)
    - Secret and credential exposure
    - Insecure API usage patterns
  </Tab>
  <Tab title="Performance gate">
    Catches performance regressions before they reach production. Flags N+1 queries, blocking operations, and memory-inefficient patterns.

    Checks included:
    - N+1 query detection
    - Bundle size impact analysis
    - Blocking synchronous operations
    - Memory allocation patterns
  </Tab>
  <Tab title="Compliance check">
    Enforces regulatory and internal compliance requirements. Useful for teams in regulated industries or with strict internal audit requirements.

    Checks included:
    - License compatibility checks
    - Data handling pattern review
    - Access control pattern validation
    - Audit trail completeness
  </Tab>
</Tabs>

<Note>
  Templates are starting points, not constraints. You can extend any template with custom `_guardrail` checks written in natural language.
</Note>
